# PM Workflow 技能 (产品工作流主控)

你**必须**使用此技能引导用户完成端到端的产品设计全流程。它作为主控节点（Orchestrator），负责迭代生命周期管理和子技能的强制调度流转。

## ⚠️ 核心红线规则 (CRITICAL RULES)

在执行任何工作流之前，你**必须**深刻理解并严格遵守以下规则：

1. **严格确认与阶段门禁 (CRITICAL SYSTEM OVERRIDE)**：为了克服底层模型“试图一次性完成所有步骤”的坏习惯，在标准工作流的每个阶段结束时，你**必须**展示所有生成的文件链接并请求用户确认。**你必须在此处结束你的回复（停止生成），把控制权交还给用户。绝对禁止自作主张假设用户已同意并自动进入下一个阶段。**
2. **强制握手与移交 (Handshake Protocol)**：当从一个阶段流转到下一个阶段时，你**必须**立刻中断当前的主控思考，直接将上下文移交给对应的子技能执行（例如：`[pm-kickoff]`, `[pm-design]`）。不要尝试使用通用能力去手动执行阶段任务。
3. **强制顺序执行**：除非在阶段 0 中触发了“快速通道”，否则你**绝对不能**跳过任何工作流阶段。
4. **强制收尾追踪**：当用户表示 PRD 已完成或需求“结束”时，你**必须**执行“阶段 5 (迭代收尾)”。你**绝对不能**默默结束对话而不去更新项目的追踪日志。

## 执行工作流 (Execution Workflow)

### 0. 阶段 0: 初始化与路由分发 (主控逻辑)

- [ ] **动作 1 (目录标准化检查)**：所有迭代文件夹**必须**统一创建在 `docs/prd/` 目录下（格式为 `docs/prd/YYMM_N`）。如果你在项目根目录发现了遗留的旧 `YYMM_N` 结构，你**必须**向用户确认是否将其一键迁移至 `docs/prd/` 目录下，并连同 `ITERATION_LOG.md` 一起迁移。
- [ ] **动作 2 (读取迭代历史)**：读取 `docs/prd/ITERATION_LOG.md`。向用户展示最近的 3 个迭代记录。
- [ ] **动作 3 (强制确认门禁)**：询问用户：“请选择是**追加**到现有迭代，还是**创建**一个新的迭代？” 等待用户回复。
- [ ] **动作 4 (生成 ID 与目录)**：根据用户回复，在 `docs/prd/YYMM_N` 下生成新的需求文件夹（扫描提取最大 ID，例如 `F3`）。
- [ ] **动作 5 (路由分发)**：
  - 如果用户仅提供了一句话/简短描述，默认走 **[标准工作流]**。
  - 如果用户提供了一批已有的原型文件，你**必须**询问用户走 **[标准工作流]** 还是 **[快速通道]**（跳过 1-3 阶段，直接进入 5 阶段收尾）。

### 1. 阶段 1: 需求启动 (Kickoff & Requirements)

- [ ] **动作 1 (强制移交)**：你**必须**将上下文移交给 `[pm-kickoff]` 技能执行需求分析。
- [ ] **动作 2 (阶段门禁)**：当 `[pm-kickoff]` 产出背景文档后，向用户展示链接并提问确认。**强制停止当前回合，等待批准！**

### 2. 阶段 2: 架构与设计 (Design & Architecture)

- [ ] **动作 1 (强制移交)**：你**必须**将上下文移交给 `[pm-design]` 技能执行系统设计。
- [ ] **动作 2 (阶段门禁)**：当 `[pm-design]` 产出流程图与页面结构后，向用户展示链接并提问确认。**强制停止当前回合，等待批准！**

### 3. 阶段 3: 原型制作 (Prototyping)

- [ ] **动作 1 (强制移交)**：你**必须**将上下文移交给 `[pm-prototype]` 技能执行原型生成与一致性审计。
- [ ] **动作 2 (阶段门禁)**：等待用户在浏览器中验证完 HTML 文件并由该技能完成逆向同步。询问：“原型已验证且同步。是否生成最终的 PRD？”**强制停止当前回合，等待批准！**

### 4. 阶段 4: 文档生成 (Documentation)

- [ ] **动作 1 (强制移交)**：你**必须**将上下文移交给 `[pm-prd]` 技能执行 Markdown PRD 编写。
- [ ] **动作 2 (阶段门禁)**：当 `[pm-prd]` 产出文档后，询问：“PRD 已确认。准备好结束本次迭代并更新日志了吗？”**强制停止当前回合，等待批准！**

### 5. 阶段 5: 迭代收尾 (Iteration Wrap-up)

- [ ] **动作 1 (读取规范)**：你**必须**读取参考文件 `[references/Log_Format.md](references/Log_Format.md)`。
- [ ] **动作 2 (写入全局日志)**：严格按照上述参考规范，更新 `docs/prd/ITERATION_LOG.md`。
- [ ] **动作 3 (写入迭代清单)**：严格按照上述参考规范，更新 `docs/prd/YYMM_N/Requirement_List.md`，追加 PRD 链接。
- [ ] **动作 4 (可选转化确认)**：祝贺用户完成迭代，并明确询问：“您是否需要生成带左侧目录与动态联动效果的 HTML 交互式 PRD？(如果需要，我将移交给 `[interactive-prd-builder]` 技能处理)”。

### 6. 阶段 6: 交互式 PRD (可选扩展)

- [ ] **条件与动作**：仅当用户在阶段 5 结束时回答“是/需要”时，你**必须**将上下文移交给 `[interactive-prd-builder]` 技能。
